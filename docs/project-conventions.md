# 项目规范

## 核心原则

本项目的交付物不只是代码功能，还包括与代码同步演进的文档和知识沉淀。

换句话说，在本项目里：

- 要完成功能
- 要补充或更新文档
- 要把实现过程中的经验、坑点、约束、排查方法沉淀到知识库

这三件事共同构成一次完整交付，缺一不可。

## 适用范围

以下工作都适用本规范：

- 新功能开发
- 解析能力扩展
- 渲染能力扩展
- 数据模型调整
- diagnostics / tests / fixtures 调整
- bug 修复
- 重构
- 行为变更或兼容性修正

## 必做要求

### 1. 功能变更必须伴随文档更新

只要代码行为发生变化，就必须同步检查是否需要更新以下内容：

- `README.md`
- 架构文档
- 模块说明文档
- 开发计划文档
- 使用说明或调试说明

判断标准不是“有没有人要求写文档”，而是“后续开发者是否会因为这次变更而需要新的说明”。

### 2. 关键经验必须沉淀到知识库

满足以下任一条件时，必须更新 `docs/knowledge-base/`：

- 解决了一个非显而易见的问题
- 发现了某个格式、库、协议、PPTX 结构的特殊约束
- 修复过程里踩过坑，未来很可能再次出现
- 形成了稳定排查路径
- 明确了某类实现的边界、语义或约定

知识库记录重点不是流水账，而是：

- 现象
- 原因
- 解决方案
- 关联文件
- 后续遇到同类问题时的排查方法

### 3. 功能、测试、文档要视为同一批交付

在本项目中，一次完整变更通常应同时考虑：

- 代码实现
- 测试验证
- 文档/知识库沉淀

不能只交代码，不补说明；也不能只写文档，不验证行为。

### 4. 文档必须贴近当前实现

文档不是抽象愿景，而应尽量对应当前仓库的真实结构、真实字段、真实文件路径、真实测试样例。

要求：

- 优先引用实际文件路径
- 优先记录当前实现中的真实字段和调用链
- 避免空泛描述
- 如果实现后续变化，文档也要同步修正

## 推荐落地方式

### 开发时

做功能时，同时问自己三个问题：

1. 这次代码改动改变了什么行为？
2. 这次改动里有什么经验值得以后复用？
3. 以后别人看到这段代码，最可能在哪个点卡住？

### 收尾时

提交前至少检查：

1. 功能是否完成
2. 测试是否覆盖或已验证
3. `docs/` 是否需要更新
4. `docs/knowledge-base/` 是否需要新增或补充记录

## 知识库目录约定

知识库统一放在 `docs/knowledge-base/` 下，按主题拆分，例如：

- `phase-0-pptx-package.md`
- `xml-parser.md`
- `text-layout.md`
- `text-bullets.md`
- `renderer-svg.md`

如无合适主题，可以新增文档，但应保持“按问题域拆分”的方式，不建议把所有经验堆到一个文件里。

## 反例

以下做法不符合本项目规范：

- 只修代码，不记录关键原因和排查路径
- 改了数据模型，但不更新相关说明
- 新增了解析能力，但没有补测试或知识库
- 遇到难点只在聊天里说清楚，没有沉淀进仓库
- 同类问题已经重复踩过，但没有形成文档约束

## 验收口径

今后评估一次任务是否“完成”，不只看功能是否可用，还要看：

- 是否有必要的测试或验证
- 是否更新了相关文档
- 是否将可复用经验沉淀到了知识库

如果功能做完了，但文档和知识库没有跟上，这次交付应视为未完全完成。
