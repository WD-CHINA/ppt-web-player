# Contributing

## 项目协作原则

本项目采用“代码功能 + 测试验证 + 文档更新 + 知识库沉淀”并行交付的方式。

这意味着一次变更是否完成，不只看代码是否跑通，还要同时看：

- 功能是否实现
- 测试是否覆盖或完成必要验证
- 文档是否同步更新
- 可复用经验是否进入 `docs/knowledge-base/`

相关总原则见：

- `README.md`
- `docs/project-conventions.md`

## 什么情况下必须补文档

只要出现以下任一情况，就必须同步检查文档：

- 新增功能
- bug 修复导致行为变化
- 数据模型变化
- 解析能力变化
- 渲染能力变化
- diagnostics、fixtures、测试策略变化
- 新的开发约束或边界被确认

优先检查这些文档是否需要更新：

- `README.md`
- `docs/project-conventions.md`
- `docs/todo.md`
- 架构文档
- 模块说明文档
- `docs/knowledge-base/*.md`

## 什么情况下必须补知识库

满足以下任一条件时，必须在 `docs/knowledge-base/` 新增或更新记录：

- 解决了一个不直观的问题
- 修复过程中踩到了容易重复出现的坑
- 发现了第三方库、PPTX 结构、OOXML 节点的特殊语义
- 形成了稳定排查路径
- 明确了实现边界、模型约束或兼容性规则

知识库内容尽量包含：

- 现象
- 原因
- 解决方案
- 关联文件
- 排查清单或复用建议

## 推荐开发流程

### 1. 开始前

先明确本次变更涉及哪一层：

- package / relationship
- XML parser
- OOXML parser
- normalized model
- layout
- renderer
- player
- app

同时思考：

- 这次改动会改变什么行为
- 是否会影响现有文档
- 是否大概率会产生新的经验记录

### 2. 实现中

建议保持这四项同步推进：

- 写功能代码
- 补或改测试
- 记录关键约束
- 判断知识库落点

不要等代码全部完成后才回忆有哪些坑，因为那样最容易遗漏细节。

### 3. 收尾时

提交前至少完成以下检查：

1. 功能完成
2. 测试通过或完成必要验证
3. 文档已更新或明确无需更新
4. 知识库已更新或明确无需更新
5. 变更说明能说清“做了什么、为什么做、有哪些约束”

## 变更收尾 Checklist

每次任务收尾时，建议逐项确认：

- [ ] 功能行为已落地
- [ ] 影响范围已确认
- [ ] 相关测试已新增、更新或执行验证
- [ ] `README.md` 是否需要更新已检查
- [ ] `docs/` 下相关说明是否需要更新已检查
- [ ] `docs/knowledge-base/` 是否需要新增或补充已检查
- [ ] 关键坑点、语义约束、排查方法已沉淀到仓库
- [ ] 最终交付可以同时说明代码、测试、文档三部分

## 对“完成”的定义

在本项目中，以下情况不应视为完全完成：

- 代码改了，但没有验证
- 功能做了，但文档没更新
- 解决了复杂问题，但经验只停留在聊天里
- 同类问题可能复发，但没有形成知识库

只有当代码、验证、文档、知识沉淀四者闭环时，才算一次完整交付。
